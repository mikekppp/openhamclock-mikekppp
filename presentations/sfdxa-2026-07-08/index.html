The map

The map is the application.

Gray-line terminator · great-circle paths · live spots · togglable layers (aurora, radar, satellites, earthquakes…)

The map is the killer feature for a DX crowd. It's NOT a static image — it's interactive. Click any DX cluster spot and it draws the great-circle path right there, with distance, bearing, grid, and sunrise/sunset at both ends. The day/night terminator is live — this room knows exactly why that matters: gray-line. When the terminator is crossing an ATNO you need, you can see it. Demo this live later by clicking a few cluster spots.

New · July 2026

We run our own DX cluster node now.

  • OHC Cluster — an independent node built for OpenHamClock, feeding the hosted site
  • Aggregates RBN skimmer spots (CW/RTTY + FT8/FT4) and human spots
  • Classic telnet interface — use it from any logger or cluster client, not just OHC
  • No more depending on the goodwill (and uptime) of public DXSpider nodes
  • Custom cluster logins hardened — reconnect storms contained, dead nodes parked
  • Self-hosters can still point at any node they like

Born after a public node sysop suggested we "build our own cluster." OK. Done.

This is the newest thing in the whole deck — shipped THIS WEEK in the July drop. The story lands well: the hosted site used to ride on public DXSpider nodes, which produced reconnect storms and login-hammering incidents when upstreams went sideways. One sysop suggested we build our own node. So we did — it ingests the Reverse Beacon Network skimmer feeds (CW/RTTY and FT8/FT4) plus human spots, and serves a classic telnet cluster interface, so you can even point your regular logging program at it. In memory of WB0OEW in the login banner.

DX target intelligence

  • Type a callsign → it becomes your DX target new
  • Real IANA timezones — DX local time is civil time, not sun time
  • Distance, bearing, grids, sunrise/sunset at both ends
Both of these are pure DXer workflow. The popup means the "who is this guy" reflex doesn't cost you a browser tab and 15 seconds — and it shows local time at the DX end, which is half of deciding whether to sit in the pileup. The DX-target lookup: type a call, and the crosshair, bearing, and distance snap to his QTH. The IANA timezone fix matters more than it sounds — longitude-based time gets places like Manila and Perth visibly wrong.
Propagation

Real numbers. Real predictions. Live.

  • FT8/FT4/CW predictions run at each mode's real decode threshold — FT8 at −19 dB, not SSB math with a fudge factor new
  • Kp tracks NOAA's 1-minute estimate — reacts to a storm in minutes, not hours new
The VOACAP piece is the part this room actually cares about. OHC runs the ITU-R P.533 HF engine itself, every 10 minutes — the band-color overlay on the map IS the prediction, and the heatmap shows per-band reliability by UTC hour for your QTH. Two July fixes worth calling out loud: (1) digital-mode predictions were dishonest before — everything ran at the SSB threshold and FT8 got a post-hoc bonus that couldn't reopen a band SSB had scored at zero. Now the engine runs at each mode's real decode threshold. Real example: Atlanta→Tokyo, 15m, noon — 0% on SSB, 67% on FT8. That matches what you actually hear. (2) The Kp index used to lag up to 3 hours; it now follows NOAA's 1-minute estimated product, so when a G-storm hits you see it immediately. If K is elevated tonight, show the aurora overlay live in the demo.

Satellites on the same map as everything else.

  • Big catalog: AO-91/92, SO-50, ISS, RS-44, FO-29, GOES, METEOR, QO-100 footprint…

DX via satellite is real DX — grids and DXCC on the birds. QO-100's footprint is fun to show even though it's not workable from Florida. The screenshot on the right is the picker — search/select any satellite from the active TLE catalog and OHC tracks it. The TLE pipeline got hardened over the last two cycles with three independent upstream sources, so the birds don't vanish when CelesTrak has a bad day.
Rig control

Click a spot. Your radio tunes.

Transports

  • USB CAT — direct, no hamlib
  • rigctld (Hamlib) · flrig
  • SmartSDR, TCI, RTL-SDR
  • Cloud relay — your shack from the road

Radios

  • Yaesu, Kenwood, Icom, Elecraft
  • FlexRadio, SunSDR, Hermes-Lite 2, ANAN
  • WSJT-X / JTDX / MSHV / JS8Call decodes plotted live on the map

Plugin-based Rig Bridge — almost any modern radio over almost any transport. Log with N3FJP? That's built in now too. new

For a DX crowd this is about pileup speed: spot appears, click, you're on frequency with the right mode while the other guy is still typing the call into his VFO. WSJT-X relay is the same idea backwards — decoded callsigns appear on the map in real time, so you can *see* the opening build. New this cycle: N3FJP integration is configured right from Settings — logged QSOs and even the call you're currently typing appear on the map (live entry preview). Specific models if asked: FT-991A/891/710/DX10/5000, TS-890/590/2000/480, IC-7300/7610/9700/705/7851, K3/K4.

Install

Pick your poison.

Zero install

Visit openhamclock.com — wizard asks for callsign + grid. Done in 30 seconds.

Docker

docker-compose up -d

Self-hosted on your LAN. API calls proxied through your backend.

Local (Mac / Linux / Win / WSL)

git clone https://github.com/accius/openhamclock.git
cd openhamclock && npm ci && npm start

Open http://localhost:3000.

Raspberry Pi kiosk

curl -fsSL https://raw.githubusercontent.com/accius/openhamclock/main/scripts/setup-pi.sh | bash -s -- --kiosk

Fullscreen Chromium on boot. Wall-display ready.

Ask the chat: who runs a Pi for ham stuff? The kiosk one-liner is the killer install — fresh Pi, paste, walk away, come back to a wall display. Windows folks: the install experience was overhauled this month — one-line PowerShell install, real updater.

Questions?

Hold here for Q&A. Common questions to be ready for: - "Does it work offline?" — mostly no; some panes do (clock, sat predictions). - "Will it run on my old Pi 3?" — yes, kiosk mode is happy on a Pi 3. - "How is it different from HamClock?" — modern web stack, plugin system, active community. Same spiritual goal. - "Which cluster node does it use?" — our own OHC Cluster node now feeds the hosted site; self-hosters can point anywhere. - "Who funds it?" — it's open source, no funder. Donations welcome but not required. Hosting on openhamclock.com is on me. - "Is my data private?" — yes; self-host if you want zero external dependencies. Otherwise, only public APIs (cluster, PSKReporter) get your callsign.
